# 给「桌面端」的对接提示词（传感器授权类型对齐）

> 用途：把这份内容整段发给桌面端的开发者或 AI 助手，用来核对两边的“授权传感器类型”清单是否一致。
> 背景：密钥后台（key-manager）生成的 license key 里编码的是传感器类型的 **value（英文标识符）**；桌面端拿到 key 后用 value 判断授权。两边只要 value 集合一致，授权就正确；label（中文显示名）和分组只影响显示。

---

## 一、术语先对齐（很重要，之前疑似在这里就对不上了）

- **value**：英文标识符，例如 `hand0205`、`smallSample`。**这是授权识别键，编码进 key、用于判断授权，绝对不能改、不能拼错、大小写敏感。**
- **label**：中文显示名，例如 `触觉手套`、`10*10小样`。仅用于界面显示，可以改。
- **group**：分组名（如 `精密`、`关怀`），仅用于界面分组显示。

> ⚠️ 注意：有的文档把列名写成「Key / Value」，其中“Key=英文标识符、Value=中文名”，方向和我们相反。请务必按上面的定义回答，避免再次错位。

## 三、请桌面端回答以下问题（这样我们才能逐条 diff）

请按下面格式回复：

**1. 桌面端实际用来判断授权的 value 清单是什么？**
请把桌面端代码里硬编码（或动态读取）的**全部 value** 用同样的表格列出来（value / 显示名 / 来源文件:行号）。不要只列“支持的”，把代码里出现的全列出来。

**2. 逐项核对（用我们上面 24 个 value 为基准）：**
- 桌面端**缺哪些**？（我们有、桌面端没有 → 这些 key 发出去会用不了）
- 桌面端**多哪些**？（桌面端有、我们清单里没有 → 后台无法生成，需要补进后台权威清单）
- value **拼写/大小写不一致**的有哪些？（例如一边 `smallSample` 一边 `smallsample` / `small_sample`）

**3. 桌面端是怎么拿到 value 的？**
- [ ] 完全硬编码在桌面端代码里（请给文件路径）
- [ ] 调后台接口动态获取（请给接口名）
- [ ] 其它（说明）

**4. 授权判断逻辑：** 桌面端是「key 里包含某 value 就解锁对应功能」，还是有别的映射/别名？有没有把多个 value 当成同一个功能、或一个 value 对应多个功能的情况？

**5. 特别确认 `smallSample`：** 桌面端这边它的 value 是不是就是 `smallSample`？显示名是不是“10*10小样”？归在哪个分组？（这个之前两边对不上，重点确认。）

## 四、对齐规则（双方达成一致后照此执行）

1. **value 以桌面端实际识别的为准**——因为 key 最终要在桌面端解锁。后台会把权威清单调整成与桌面端 value 完全一致。
2. 桌面端**多出**的 value → 补进后台 `server/db.ts` 的 `DEFAULT_SENSOR_TYPES`，后台才能生成对应 key。
3. 后台**多出**、桌面端不认的 value → 从后台清单停用（避免发出用不了的 key）。
4. label / 分组以后端这份为准做显示统一（不影响授权）。
5. 任何**新增类型**：必须同时在桌面端代码 + 后台 `DEFAULT_SENSOR_TYPES` 两边登记，缺一不可。

> 请桌面端把第三节的回答发回来，我方据此更新后台权威清单并重新部署，然后再复核一轮。
